COPY PROMPT

三段式需求 Prompt

我想做一个「App Store 评论分析」Chrome 浏览器插件。 背景：经常需要看 App Store 评论，手动翻评论和整理差评很耗时间。 用户：内部业务同学，不需要登录注册。 目标：粘贴 App Store 应用链接，自动抓取评论并展示。 约束：manifest v3；用公开 RSS/JSON 接口；不接后端，本地保存即可。 验收：能输入链接、选国家、抓评论；显示评分/标题/正文/作者；顶部显示总数和平均评分。
这页不用逐字讲完，可以说“这段之后会作为小抄给大家”。

FEEDBACK

反馈修改：现象 + 预期 + 要求

现象： 点击“抓取”后，按钮没有任何变化，用户不知道是不是在加载。 预期： 按钮应该变成“抓取中...”，并且暂时不能重复点击。 修改要求： 只改按钮加载状态和防重复点击，不要改评论列表和接口逻辑。
不要只说“不对”“不好用”。这套模板适合功能、界面、文案和流程。
UI FIX

不满意界面？用截图，别用嘴

  1. 贴截图，让 AI 看到现状。
  2. 指出要改哪里，不要抽象说“高级”。
  3. 一次只说几个具体点，布局不要动。
一张截图加具体指令，胜过十句形容词。
BACKUP

敢大胆改，是因为随时能回退

v1能跑就复制一份
Git进阶版本备份

改乱了不可怕，能跑的版本才值钱。

给安全感。新手最怕改坏，先教最简单的复制文件夹，后面再提 Git。
DEBUG

报错不是坏事，报错是线索

插件白屏 / 点了没反应：右键小窗口 → 检查 → 控制台 → 复制红色报错。

报错信息是给 AI 的线索，不是给你看懂的。你只负责搬运。
FIRST AID

急救三板斧

  1. 贴完整报错，不要自己转述。
  2. 同一问题超过 2 次不对，新开会话。
  3. 缩小范围：只改抓取，不动其它。
不要不停说“还是不行”。换思路：贴报错、开新会话、缩小范围。
ERROR PROMPT

修报错 Prompt

现象： [我做了什么操作，然后发生了什么] 完整报错： [把红色报错整段粘进来] 预期： [我原本期待发生什么] 修改要求： 请先判断原因，再只修改相关文件。不要顺手重构其它功能。
这是排错页最实用的模板，可以让大家拍照带走。

RESTART

什么时候该推倒重来

SIGNAL
一个功能改了五六次还不对
旧对话可能被错误尝试污染。
ACTION
退回上个能跑版本
新开会话，重新、更清楚地说一遍。
沉没成本不值得心疼。能跑的版本才值钱。
PRODUCTIZE

从玩具到能给别人用

  1. 链接填错：友好提示，不要报错。
  2. 没有评论：告诉用户原因。
  3. 正在抓取：显示加载中，禁止重复点击。
  4. 接口失败：提示重试，不要白屏。
边界情况是“我自己能用”和“别人能用”的分水岭。
SMOKE TEST

让 AI 帮你设计测试清单

请帮我为这个插件设计一份最基础的冒烟测试清单。 要求： - 覆盖正常流程、错误输入、无评论、网络失败、重复点击 - 每条测试写清楚：操作步骤、预期结果、是否通过 - 先不要写复杂自动化测试，用我能手动照着测的方式
把专业做法翻译成新手能用的形式：连测试都能让 AI 帮你写。
SECURITY

数据安全与隐私：上线前必查

  1. 不要把密码、密钥、API Key 写死在代码里。
  2. 评论和分析结果，不要随手贴到公网工具。
  3. 拿不准，就找懂技术的同事看权限和数据。
尤其 Part 8 接 AI 时要提醒。能跑不等于能安全上线。

AI APP

抓评论和做统计是普通应用；让 AI 读懂差评、归纳问题、生成建议，就是 AI 应用。

WHERE AI FITS

一句话判断：哪里该塞 AI

USE AI
理解 / 生成 / 判断
总结、归类、翻译、回复、提建议。
NO AI
计算 / 排序 / 统计
数星星、算平均分、统计词频，普通代码更稳。
防止“什么都用 AI”。AI 用在理解、生成、判断；普通逻辑用于确定性计算。
RELEASE

方式怎么做适合

铁律：先让同事能用起来，再慢慢优化。内部分享一定要附使用说明。
PRINCIPLES

迭代节奏五原则

  1. 小步：一次一个功能。
  2. 可运行：每步都保持能装、能用。
  3. 常验收：按最初标准一条条对。
  4. 敢回退：乱了就回上一个能跑版本。
  5. 早发布：别在自己电脑上磨太久。
这是 vibe coding 的手感。任何应用都适用。
